# OTA Updates Plan — GitHub-Based Install and Update Infrastructure

**File:** `ota_updates_8d9e0f1a.plan.md`
**Status:** Draft — awaiting design review.

---

## Goal

Use GitHub as the complete infrastructure for:
1. **Initial installs** — users download a signed binary from the GitHub
   Releases page.
2. **App updates** — the installed app checks a GitHub Pages endpoint and
   installs new versions automatically (user-confirmed).
3. **Rules updates** — JSON rule files (web import, board profiles) are updated
   by committing to this repo; no app release needed.

No external server, no hosting costs, no infrastructure to maintain.

## One-time setup (do this once before first release)

### Step 1 — Enable GitHub Pages

1. Go to `https://github.com/x-squared/X2Chess/settings/pages`
2. Under **Source**, choose **GitHub Actions**
3. Click **Save**

GitHub Pages will be served at `https://x-squared.github.io/X2Chess/`.

### Step 2 — Generate a signing key

The signing key is what proves an update binary genuinely came from you.
Tauri refuses to install updates that are unsigned or signed with the wrong key.

Run this once on your machine (requires the Tauri CLI):

```bash
cd frontend
npx tauri signer generate -w ~/.tauri/x2chess.key
```

This prints two things:
- **Private key** — keep this secret; goes into GitHub as a CI secret.
- **Public key** — safe to commit; goes into `tauri.conf.json`.

### Step 3 — Add secrets to GitHub

Go to `https://github.com/x-squared/X2Chess/settings/secrets/actions` and
add two repository secrets:

| Secret name | Value |
|---|---|
| `TAURI_SIGNING_PRIVATE_KEY` | The private key output from Step 2 |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | The password you chose (or empty string) |

### Step 4 — Add the public key to tauri.conf.json

In `frontend/src-tauri/tauri.conf.json`, add the updater plugin config:

```json
{
  "plugins": {
    "updater": {
      "pubkey": "<paste public key from Step 2 here>",
      "endpoints": [
        "https://x-squared.github.io/X2Chess/latest.json"
      ],
      "dialog": false
    }
  }
}
```

`"dialog": false` means the app handles the update UI itself rather than
showing the OS default dialog.

### Step 5 — Add the updater plugin to Cargo.toml

In `frontend/src-tauri/Cargo.toml`:

```toml
[dependencies]
tauri-plugin-updater = "2"
```

And register it in `frontend/src-tauri/src/main.rs` (or `lib.rs`):

```rust
tauri::Builder::default()
    .plugin(tauri_plugin_updater::Builder::new().build())
    // ... other plugins
```

## Repository structure

After setup, the repo gains these directories:

```
.github/
└── workflows/
    ├── ci.yml          # Runs on every push + PR: typecheck, test
    └── release.yml     # Runs when you push a tag like v0.2.0

update-server/
├── README.md           # Explains this directory
└── rules/
    ├── manifest.json   # Rules version index (edit this when bumping rules)
    └── web-import/
        └── v1.json     # Web import rule definitions
```

`latest.json` is **not** committed — it is generated by `release.yml` on each
release and deployed to GitHub Pages automatically.

## Workflow: CI (`ci.yml`)

Runs on every push and every pull request. Catches type errors and broken tests
before they reach main.

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
          cache-dependency-path: frontend/package-lock.json

      - name: Install frontend dependencies
        run: npm ci
        working-directory: frontend

      - name: Type-check
        run: npm run typecheck
        working-directory: frontend

      - name: Test
        run: npm test
        working-directory: frontend
```

## Workflow: Release (`release.yml`)

Triggered by pushing a version tag like `v0.2.0`. Builds the app for all four
platforms, signs each binary, publishes a GitHub Release, generates
`latest.json`, and deploys everything to GitHub Pages.

The heavy lifting is done by the official `tauri-apps/tauri-action@v0`.

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  publish:
    permissions:
      contents: write   # needed to create GitHub Release

    strategy:
      fail-fast: false
      matrix:
        include:
          - platform: macos-latest       # Apple Silicon (arm64)
            args: --target aarch64-apple-darwin
          - platform: macos-13           # Intel (x86_64)
            args: --target x86_64-apple-darwin
          - platform: ubuntu-22.04       # Linux x64
            args: ''
          - platform: windows-latest     # Windows x64
            args: ''

    runs-on: ${{ matrix.platform }}

    steps:
      - uses: actions/checkout@v4

      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
          cache-dependency-path: frontend/package-lock.json

      - name: Install Rust stable
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.platform == 'macos-latest' && 'aarch64-apple-darwin' || '' }}

      - name: Install Linux dependencies
        if: matrix.platform == 'ubuntu-22.04'
        run: |
          sudo apt-get update
          sudo apt-get install -y libwebkit2gtk-4.1-dev libappindicator3-dev \
            librsvg2-dev patchelf

      - name: Install frontend dependencies
        run: npm ci
        working-directory: frontend

      - name: Build and publish
        uses: tauri-apps/tauri-action@v0
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
          TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
        with:
          projectPath: frontend
          tagName: ${{ github.ref_name }}
          releaseName: X2Chess ${{ github.ref_name }}
          releaseBody: See the full changelog in CHANGELOG.md
          releaseDraft: true      # review before publishing
          prerelease: false
          args: ${{ matrix.args }}

  deploy-pages:
    needs: publish
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write

    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - uses: actions/checkout@v4

      - name: Download latest.json from release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          TAG=${{ github.ref_name }}
          gh release download $TAG --pattern "latest.json" --dir update-server/

      - name: Upload Pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: update-server/

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## How to make a release (step by step)

1. **Update the version number** in two places:
   - `frontend/src-tauri/tauri.conf.json` → `"version": "0.2.0"`
   - `frontend/src-tauri/Cargo.toml` → `version = "0.2.0"`

2. **Commit** those changes:
   ```bash
   git add frontend/src-tauri/tauri.conf.json frontend/src-tauri/Cargo.toml
   git commit -m "chore: bump version to 0.2.0"
   git push
   ```

3. **Push a tag** — this is the trigger for the release workflow:
   ```bash
   git tag v0.2.0
   git push origin v0.2.0
   ```

4. **Watch the workflow** at `https://github.com/x-squared/X2Chess/actions`.
   It will take 15–25 minutes to build all four platform binaries.

5. When the workflow succeeds, go to
   `https://github.com/x-squared/X2Chess/releases` and find the new **draft
   release**. Review it, then click **Publish release**.

6. `latest.json` will be live at
   `https://x-squared.github.io/X2Chess/latest.json` within a minute.
   Installed apps will pick it up on their next startup check.

## How to update rules (no release needed)

To add or change a web import rule, a board profile, or any other rule:

1. Edit the relevant file in `update-server/rules/`.
2. Bump the version number in `update-server/rules/manifest.json`.
3. Commit and push to `main`.
4. The deploy-pages workflow (or a separate `rules.yml` workflow) deploys to
   Pages automatically within a minute or two.
5. Running apps pick up the new rules on their next startup.

This is a data-only update — no binary is built, no signing required.

---

## Channel 2 — Rules server files

`update-server/rules/manifest.json` tracks versions of each rule set:

```json
{
  "version": 1,
  "updated": "2026-03-23",
  "rules": {
    "webImport": {
      "version": 1,
      "url": "rules/web-import/v1.json"
    },
    "boardProfiles": {
      "version": 1,
      "url": "rules/board-profiles/v1.json"
    }
  }
}
```

When the app starts it fetches this manifest, compares versions against what
is cached locally, and downloads any updated rule files. Failures are silent —
cached rules are used as fallback. No restart required.

## Security notes

- **Binaries are signed** with Ed25519. Tauri verifies the signature before
  installing. A compromised GitHub Release page cannot deliver a malicious
  update because the signature check will fail.
- **Rules are data only**. The rules server cannot deliver executable code.
  See `ota_updates_8d9e0f1a.plan.md` for why dynamic code loading is
  intentionally excluded.
- **The private key never touches CI runners**. It is stored as a GitHub
  Actions secret and accessed only as an environment variable during the build.
  Never commit the private key to the repository.
